# CodeQuest 源码闯关阅读器 — 设计文档

日期:2025-06-10
状态:已确认

## 1. 产品概述

一个本地运行的 Web 应用:导入本地代码仓库 → AI 自动分析并生成「游戏闯关课程」→ 用户以复古像素游戏(金山打字通时代风格)的方式闯关,循序渐进理解项目:**架构总览 → 核心模块 → 业务流程 → 设计决策**。仓库有新提交时,基于 git diff **增量更新**关卡,不全量重建、不丢进度。

启动方式:`npx code-quest` → 本地起服务 → 自动打开浏览器。

## 2. 总体架构

```
┌────────────────────── 浏览器 (React + Vite + TS) ──────────────────────┐
│  导入向导    闯关地图(像素风)    关卡页(代码浏览器+任务面板)    成就/证书  │
└────────────────────────────┬──────────────────────────────────────────┘
                    HTTP + SSE(生成进度实时推送)
┌────────────────────────────┴──────────────────────────────────────────┐
│                     本地 Node 服务 (Fastify + TS)                      │
│                                                                        │
│  RepoScanner        LevelGenerator       LLMProvider      ProgressStore│
│  目录树/git信息       生成管线/增量更新      claude CLI⇄API    进度/XP/徽章 │
└────────────────────────────┬───────────────────────────────────────────┘
                             ▼
              ~/.code-quest/<projectId>/
                ├─ project.json   仓库路径、commit 锚点、元信息
                ├─ course.json    课程大纲:章节 → 关卡
                ├─ levels/*.json  每关的任务、题目、代码引用
                └─ progress.json  闯关进度、XP、徽章
```

设计要点:

- 仓库结构为 monorepo,两个包:`packages/server`(Fastify 后端)与 `packages/web`(React 前端),共享类型包 `packages/shared`(关卡/任务/进度的 TS 类型 + zod schema)。
- 数据全部为 JSON 文件(无数据库),可读、可手工修正、可整目录拷贝分享。
- 关卡不内嵌代码副本,只存 `{ file, startLine, endLine, contentHash }` 引用,游玩时实时从仓库读取;hash 不匹配则该任务标记「源码已变化」。
- **LLMProvider 可插拔**:接口仅 `generate(prompt, schema) → Promise<JSON>`。
  - 实现一(优先):本机 `claude` CLI,headless 模式 `claude -p --output-format json`,可让 Claude 自行探索仓库。
  - 实现二(回退):Anthropic API SDK,需 `ANTHROPIC_API_KEY`,后端负责拼装文件内容上下文,tool-use 强制 schema。
  - 启动时自动探测可用项,导入向导中显示当前使用的 Provider。

## 3. 课程与关卡体系

```
课程 = 章节×4~6(地图区域) → 关卡×3~6/章(地图节点,顺序解锁) → 任务×3~8/关
第1章 初入江湖:项目定位、目录全景、技术栈、启动入口
第2章 架构探秘:分层与模块划分、核心抽象、依赖关系
第3章 业务征途:主业务流程逐条走通(每流程 1-2 关)
第4章 设计之道:关键设计决策、模式运用、为什么这么写
(大仓库由测绘阶段自动增加章节;小仓库可合并)
```

四种任务类型:

| 类型 | 玩法 | 判定 |
|---|---|---|
| `quiz` | 剧情式讲解 + 指定代码片段阅读 → 单选/多选/判断 | 选项比对 |
| `treasure-hunt` | 「找到处理 XX 的函数,在代码浏览器点击那一行」 | 文件 + 行范围命中 |
| `call-chain` | 乱序函数调用卡片,拖拽排成正确执行顺序 | 序列比对 |
| `code-fill` / `code-type` | 挖空关键代码补全;或逐字临摹(金山打字通式),实时对错高亮 + WPM/准确率统计 | 文本比对(忽略空白)/ 逐字符比对 |

## 4. AI 生成管线(LevelGenerator)

1. **测绘阶段**(1 次 LLM 调用):分析仓库 → 课程大纲 JSON(章节、每关主题、每关核心文件清单)。强 schema 校验,失败附校验错误自动重试。
2. **出题阶段**(每关 1 次调用,串行带并发上限):输入 = 关卡主题 + 相关文件内容;输出 = 任务数组。后端逐一校验任务的代码引用真实存在 + 内容 hash 一致;引用造假的任务丢弃,数量不足则补生成。
3. **落盘即玩**:每关生成完即写 `levels/<id>.json`,SSE 通知前端点亮地图节点;用户不必等全部生成完即可开始第一关。
4. 所有 LLM 输出经 zod schema 校验;失败重试 2 次(附校验错误);仍失败 → 该关标记「生成失败,可单独重试」,不阻塞其他关。
5. 生成状态落盘;服务重启/页面关闭后重新打开可续跑未完成的关,已完成的不重新生成。

## 5. 增量更新引擎(核心需求)

```
检测:打开地图时对比 HEAD 与锚点 commit → 不同则弹「世界发生了变化!」公告
用户点「更新关卡」:
1. git diff --name-status <锚点>..HEAD
2. 影响分析(每关存有引用文件集合):
   · 引用文件被修改 → 关卡标记 stale → 送 LLM「修订」(输入旧关卡 JSON + 相关 diff,要求最小改动)
   · 引用文件被删除 → 关卡作废,该关进度归档
   · 新增文件/模块达到阈值 → 测绘阶段轻量重跑,只允许「追加」新关卡,不允许改动旧大纲
3. 全部新/修订关卡先写临时目录,成功后原子替换并更新锚点 commit
未受影响的关卡与全部已有进度原样保留。
```

## 6. 游戏化前端

- **视觉**:NES.css + 像素字体(Press Start 2P / 中文 Zpix),CRT 扫描线滤镜可开关。代码浏览器例外:现代等宽字体 + Shiki 语法高亮,保证阅读体验。
- **导入向导**:像素风「新游戏」界面;路径输入 → Provider 探测 → 「世界生成中…」进度页,边生成边点亮章节。
- **闯关地图(主界面)**:横向卷轴;章节 = 不同配色像素场景区域;关卡节点:已通关点亮 / 当前关闪烁 / 未解锁灰色上锁。顶栏:等级、XP 条、徽章栏。
- **关卡页(左右分栏)**:
  - 左:代码浏览器 —— 文件树 + Shiki 高亮源码;寻宝任务在此点击作答;任务可触发自动跳转定位。
  - 右:任务面板 —— 像素 NPC 向导对话式讲解 → 任务交互区(选择/拖拽/填空/临摹)→ 即时判定反馈。
- **数值与反馈**:答对 +XP(基础分 × 连击系数)、8-bit 音效(Web Audio 实时合成,无音频素材依赖);答错扣一颗心 + 给提示;心用完本关重来(任务顺序不变)。通关评级 S/A/B/C(准确率 + 连击),章节通关像素烟花。
- **成就体系**:徽章墙(示例:「初窥门径」通关第一章、「一气呵成」满连击过关、「考古学家」累计阅读 50 个文件);等级称号:见习读者 → 代码学徒 → 架构行者 → 源码宗师。全课程通关生成可截图分享的通关证书页(项目名 + 完成度 + 总评级)。
- **更新提示**:检测到新 commit 时地图弹像素公告,受影响关卡节点显示「!」标记。
- **前端技术**:React + Vite + TypeScript、Zustand 状态管理、SSE 接收生成进度、dnd-kit 拖拽排序、自研打字临摹组件(逐字符 diff、WPM/准确率)。

## 7. 错误处理

| 场景 | 处理 |
|---|---|
| 路径不存在 / 非 git 仓库 | 导入向导即时校验;非 git 仓库允许降级导入(失去增量更新,UI 明示) |
| 无 claude CLI 且无 API key | 导入前置检查,给出两种配置方式引导 |
| LLM 输出 schema 校验失败 | 自动重试 2 次(附校验错误);仍失败 → 该关标记可重试,不阻塞其他关 |
| 生成中断 / 服务重启 | 生成状态落盘,重开续跑 |
| 本地未提交修改致 hash 不匹配 | 任务降级为只读展示并提示「源码已变化」,关卡仍可通过 |
| 增量更新中断 | 临时目录 + 原子替换,不存在半更新状态 |

## 8. 测试策略

- **单元测试(Vitest)**:四类任务判定纯函数、XP/连击/评级计算、diff → 关卡影响分析、代码引用校验器、schema 校验。
- **集成测试**:内置 fixture 小型 git 仓库 + Mock LLMProvider(返回预设 JSON),跑通「导入 → 生成 → 游玩 → commit 变更 → 增量更新」全链路;CLI/API 两个 Provider 实现做契约测试。
- **前端测试**:判定与状态逻辑抽为纯函数测试;Testing Library 覆盖答题提交、拖拽排序、打字判定等关键交互。
- 真实 LLM 调用不进 CI,保留手动冒烟脚本。

## 9. 范围边界(第一版不做)

- 公网部署 / 多用户 / 鉴权
- Git URL 克隆导入、zip 上传(仅本地路径)
- 关卡包导出分享(架构上已留可能性:整目录拷贝即可)
- 移动端适配
